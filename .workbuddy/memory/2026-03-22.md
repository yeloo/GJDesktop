#### 位置快照保存：

```cpp
DesktopLayoutSnapshot DesktopAutoArrangeService::createPositionSnapshot(
    const std::vector<ArrangeableDesktopIcon>& icons
) {
    DesktopLayoutSnapshot snapshot;
    snapshot.positions.reserve(icons.size());

    // 获取当前时间戳
    time_t now = time(nullptr);
    char timeBuffer[64];
    strftime(timeBuffer, sizeof(timeBuffer), "%Y-%m-%d %H:%M:%S", localtime(&now));
    snapshot.timestamp = timeBuffer;

    // 保存每个图标的位置
    for (const auto& icon : icons) {
        if (!icon.identity.parsingName.empty()) {
            snapshot.positions.push_back(DesktopIconPositionSnapshot(
                icon.identity.displayName,
                icon.identity.parsingName,
                icon.currentPosition,
                icon.category
            ));
        }
    }

    snapshot.totalCount = snapshot.positions.size();
    return snapshot;
}
```

## 10:31 修复 GitHub Actions 编译错误（未定义类型和结构体）

### 问题分析：

1. **QTextEdit 未定义类型错误**
   - 错误位置：`main_window.cpp:184,185,186,568`
   - 根因：`main_window.h` 中只对 `QTextEdit` 做了前向声明，但成员函数中使用了 `setText()` 等方法
   - MOC 处理时需要完整的类定义才能生成正确的元对象代码

2. **OrganizePlan 和 LayoutPlanResult 未定义结构体错误**
   - 错误位置：`main_window.h:179,182`
   - 根因：这两个结构体作为成员变量（值类型）声明，但只做了前向声明
   - MOC 处理和编译时都需要完整的结构体定义来计算成员大小

### 修复方案：

1. **main_window.h** - 添加完整头文件包含
   ```cpp
   // 修复前
   class QTextEdit;  // 仅前向声明

   // 修复后
   #include <QTextEdit>  // 完整包含
   ```

2. **main_window.h** - 成员变量改为智能指针
   ```cpp
   // 修复前
   ccdesk::core::OrganizePlan m_lastFilePlan;
   ccdesk::core::LayoutPlanResult m_lastLayoutPlan;

   // 修复后
   std::unique_ptr<ccdesk::core::OrganizePlan> m_lastFilePlan;
   std::unique_ptr<ccdesk::core::LayoutPlanResult> m_lastLayoutPlan;
   ```

3. **main_window.cpp** - 更新所有使用处
   - 将 `m_lastFilePlan.xxx` 改为 `m_lastFilePlan->xxx`
   - 将 `m_lastLayoutPlan.xxx` 改为 `m_lastLayoutPlan->xxx`
   - 添加智能指针空值检查：`if (m_lastFilePlan && ...)`

### 涉及的代码位置：

- `onGenerateFileOrganizePlan()`: 生成规划时使用 `make_unique` 创建
- `onRefreshFileResults()`: 刷新时检查智能指针是否为空
- `onGenerateLayoutPlan()`: 生成规划时使用 `make_unique` 创建
- `onRefreshLayoutResults()`: 刷新时检查智能指针是否为空
- `showFileOrganizeResults()`: 显示结果时解引用访问
- `showLayoutResults()`: 显示结果时解引用访问

### 技术细节：

**为什么使用智能指针而不是值类型？**
- 避免在头文件中需要完整的结构体定义
- 降低编译依赖，加快编译速度
- 符合现代 C++ 最佳实践
- MOC 处理时只需要指针类型，不需要完整定义

**为什么不能用前向声明 + 值类型？**
- 值类型需要编译器知道对象大小来分配内存
- 前向声明只告诉编译器"这个类型存在"，不知道大小
- MOC 生成元对象代码时需要完整信息

#### 1. **新增SnapshotManager快照管理器（snapshot_manager.h/.cpp）**

功能：
- **快照持久化存储**：将执行前的桌面图标位置保存到 JSON 文件
- **保存快照**：`saveSnapshot()` - 序列化快照为 JSON 并写入磁盘

- **清除快照**：`clearAllSnapshots()` - 删除所有快照文件

快照格式（JSON）：
```json
{
  "timestamp": "2026-03-22 11:24:00",
  "totalCount": 15,
  "positions": [
    {
      "displayName": "文档.txt",
      "parsingName": "C:\\Users\\...\\文档.txt",
      "originalPosition": { "x": 100, "y": 200 },
      "category": "文档"

  ]
```

## 11:42 修复编译错误：unique_ptr 需要完整类定义

### 问题分析：

GitHub Actions 编译失败，错误位置：`app_manager.cpp` 编译时
```
error C2027: use of undefined type 'ccdesk::core::SnapshotManager'
error C2338: static_assert failed: 'can't delete an incomplete type'
warning C4150: deletion of pointer to incomplete type; no destructor called
```

根因：
- `DesktopAutoArrangeService` 的析构函数是 `= default`
- 析构函数需要调用 `std::unique_ptr<SnapshotManager>` 的析构函数
- `std::unique_ptr` 的析构函数需要调用 `delete` 释放 `SnapshotManager` 对象
- `delete` 操作需要知道 `SnapshotManager` 的完整类定义（析构函数、对象大小等）
- 之前只有前向声明 `class SnapshotManager;`，不足以支持 `delete` 操作

调用链：
```
app_manager.cpp 包含 desktop_auto_arrange_service.h
  → 实例化 DesktopAutoArrangeService 对象
    → 调用析构函数（编译器生成的默认析构）
      → 调用 std::unique_ptr<SnapshotManager>::~unique_ptr()
        → 调用 std::default_delete<SnapshotManager>::operator()()
          → 执行 delete snapshotManagerPtr;
            ❌ 需要完整类定义！
```

### 修复方案：

**desktop_auto_arrange_service.h** - 删除前向声明，包含完整头文件
```cpp
// 修复前
// 前向声明 SnapshotManager
namespace ccdesk::core {
class SnapshotManager;
}

// 修复后
#include "snapshot_manager.h"  // 提供完整类定义
```

### 技术细节：

**为什么前向声明不够？**

对于 `std::unique_ptr<T>`：
- ✅ 前向声明足够：声明指针类型、作为函数参数/返回值
- ❌ 前向声明不够：析构函数、`std::make_unique<T>()`、调用 `T` 的方法

**编译器需要完整定义的场景：**
1. **析构函数**：需要调用 `T` 的析构函数
2. **默认删除器**：`std::default_delete<T>` 需要执行 `delete` 操作
3. **自定义删除器**：如果删除器访问 `T` 的成员

**Pimpl 模式 vs 智能指针：**
- **Pimpl 模式**：可以在头文件中用前向声明，因为析构在 cpp 中实现
- **智能指针成员**：如果析构是 `= default`，必须在头文件中包含完整定义
- **智能指针成员**：如果析构在 cpp 中实现，可以用前向声明

**本案例的约束：**
- `DesktopAutoArrangeService` 的析构是 `~DesktopAutoArrangeService() = default;`
- 在头文件中定义为 `= default`
- 编译器会在头文件中内联生成析构代码
- 因此必须在头文件中包含完整类定义

**替代方案（如果必须降低编译依赖）：**
```cpp
// desktop_auto_arrange_service.h
~DesktopAutoArrangeService();  // 不使用 = default

// desktop_auto_arrange_service.cpp
DesktopAutoArrangeService::~DesktopAutoArrangeService() = default;
```
然后可以在头文件中保持前向声明，在 cpp 中包含完整头文件。

但考虑到：
- `SnapshotManager` 头文件已经很小（2.45 KB）
- 其他头文件已经包含了很多依赖
- 当前方案更直接、更易维护

所以选择在头文件中包含完整定义。
